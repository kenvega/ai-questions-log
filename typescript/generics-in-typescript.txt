question
  what are generics?
  can you explain their syntax?
  why there were created?
  where they are useful?
  do they have an alternative?
  if we didn't have them what problems would we have?

response
  generics are "type variables"

  They let you write code that works with many types
    while still preserving the exact type information
      example

        function identity<T>(value: T): T {
          return value;
        }

        const a = identity("hello"); // string
        const b = identity(123);     // number

        T is not a real runtime variable.
          It is a placeholder for a type.

        TypeScript’s docs describe generics as
          a way to build reusable components that
            work over a variety of types rather than only one type.

  Syntax
    in the function
      function identity<T>(value: T): T {
        return value;
      }

      The <T> part means:
        I am creating a type variable called T
          and I want to use it inside this function signature

      the function can be read like this:
        function identity<T>(value: T): T
        //       name     ↑       ↑     ↑
        //                |       |     return type uses T
        //                |       parameter uses T
        //                declare T here

          without <T>, this would not work:
            function identity(value: T): T {
              return value;
            }

            TypeScript would ask:
              Cannot find name 'T'
                Because T was never declared.

    Think of <T> like declaring a variable

      This is similar to regular function parameters.

        example:
          You cannot do this:
            function double(): number {
              return x * 2;
            }

          Because x was never declared.

          You need:
            function double(x: number): number {
              return x * 2;
            }

        Same idea with generic types.
          You cannot do this:
            function identity(value: T): T {
              return value;
            }

          Because T was never declared.

          You need:
            function identity<T>(value: T): T {
              return value;
            }

            <T> declares T

    Why is it placed after the function name?

      Because it describes the function itself
        function identity<T>(value: T): T

      means:
        identity is a generic function with one type parameter called T

      When you call it, TypeScript usually figures out T automatically:
        const a = identity("hello");
        // TypeScript infers T = string

        const b = identity(123);
        // TypeScript infers T = number

      So these are equivalent:
        identity<string>("hello");
        identity("hello");

      Most of the time, you do not manually write the type
        when calling it because TypeScript infers it

    Why not just infer T from (value: T): T?
      In theory, TypeScript could have been designed that way.
        But TypeScript requires generic type variables to be declared explicitly.
        That makes the code clearer and avoids ambiguity
          especially when you have multiple generic types, constraints, or defaults.

          example:
            function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
              return obj[key];
            }

          Here the generic declaration is doing more:
            <T, K extends keyof T>

          It says:
            There are two type variables, T and K, and K must be one of the keys of T.

          example:
            const user = {
              id: 1,
              name: "Ken",
            };

            const name = getProperty(user, "name"); // string
            const id = getProperty(user, "id");     // number

            getProperty(user, "email"); // Error

          Here, <T, K extends keyof T> is not repetitive at all.
            It defines the relationship between the types.

    Simple mental model
      function identity<T>(value: T): T

      means
        Create a generic function named identity.
        It has a type variable called T.
        The parameter is of type T, and the return value is also of type T

  Why were generics created?

    They solve this problem:
      “I want reusable code, but I do not want to lose type safety.”

    Without generics, you often end up choosing between duplication and weak types.
      example
        without generics code becomes repetitive:
          function getFirstString(items: string[]): string {
            return items[0];
          }

          function getFirstNumber(items: number[]): number {
            return items[0];
          }

        you could use any:
          function getFirst(items: any[]): any {
            return items[0];
          }

          const value = getFirst(["a", "b"]);
          // value is any, so TypeScript no longer protects you
          value.toFixed(); // no error at compile time, but crashes at runtime

        generics fix both problems:
          function getFirst<T>(items: T[]): T {
            return items[0];
          }

          const name = getFirst(["John", "Doe"]); // string
          const age = getFirst([20, 30]);        // number

          name.toUpperCase(); // OK
          age.toFixed();      // OK

        The function is reusable, but TypeScript still knows what comes out.

  Where are generics useful?

    They are useful anywhere the input type and output type are related.

      1. Arrays

          You already use generics when you write this:
            const names: Array<string> = ["John", "Doe"];
            const numbers: Array<number> = [1, 2, 3];

          Array<string> means “an array whose items are strings.”
            TypeScript’s docs use arrays as a common example of generics.
