question
  what are types and interfaces?
  seems they overlap in features which makes it a bit more complicated than needed. why is that?
  why interfaces were created for?
  why types were created for?
  which one was created first?
  if both were created at the same time
    then they had a reason to exist even though they overlap?
  how are they similar and how they are different?
  in which cases devs might use them not in the intended way they need to?

response
  The cleanest separation is this:
    interface = name an object contract
    type = name any type expression

    The confusion happens because an object shape is also a type expression
      so both can describe it

    Interfaces were created mainly to model JavaScript objects and APIs.

  Which came first?

    Interfaces came first.
      TypeScript’s first public release was TypeScript 0.8 in October 2012

    Type aliases came later, in TypeScript 1.4.

  Why were interfaces created?

    Interfaces were created mainly to model JavaScript objects and APIs.

    JavaScript uses objects everywhere:
      createButton({
        text: "Submit",
        color: "blue",
      });

    TypeScript needed a way to say:
      interface ButtonSettings {
        text: string;
        color?: string;
      }

      That means:
        “Anything with this shape is acceptable.”

  Why were type aliases created?

    type was created later because
      TypeScript needed a way to give a name to any type expression
      not only object contracts

    examples
      type ID = string | number;
      type Callback = () => void;
      type Status = "idle" | "loading" | "success" | "error";
      type Point = [number, number];

    An interface cannot directly represent that.

    type was not originally “another interface.” It was:

      “I have a type expression. I want to reuse it by name.”

        Then this also became possible:

        type User = {
          id: number;
          name: string;
        };

        And now it overlaps with interface.

  Why can both describe objects?

    Because TypeScript is structurally typed.

    This means TypeScript mostly does not care where a type came from. It cares what shape it has.

    These two are effectively the same shape:

      interface User {
        id: number;
        name: string;
      }

      type User = {
        id: number;
        name: string;
      };

      TypeScript sees both as:
        An object with id: number and name: string

      So the overlap exists because:
        interface names object shapes
        type names any type expression
        An object shape is one kind of type expression
        They overlap because an object shape is also a type expression
          interface User { ... } // Here is an object contract
          type User = { ... } // Here is a name for this object type expression

          Same result, different original purpose

  How they are similar

    Both disappear at runtime.
      They only help TypeScript check your code before or during compilation.

    Both type and interface can describe objects:
      type Product = {
        id: number;
        price: number;
      };

      interface Product {
        id: number;
        price: number;
      }

    Both can describe function shapes:
      type Greet = (name: string) => string;

      interface Greet {
        (name: string): string;
      }

    Both can be extended/composed:
      type Admin = User & {
        role: "admin";
      };

      interface Admin extends User {
        role: "admin";
      }

  Main difference

    The biggest practical difference is:

      - interface is mainly for describing object/class shapes.

      - type is more flexible and can represent many kinds of types, not just object shapes.

      For example, type can do this:
        type ID = string | number;

      An interface cannot directly represent a union like that.

      You can also do:
        type Status = "loading" | "success" | "error";

      This is one of the most common uses for type.

    interface can be reopened; type cannot

      Interfaces can be declared multiple times and TypeScript will merge them:
        example
          interface User {
            id: number;
          }

          interface User {
            name: string;
          }

          const user: User = {
            id: 1,
            name: "Ken",
          };

        This works because the two User interfaces merge.

        But this does not work with type:
          type User = {
            id: number;
          };

          type User = {
            name: string;
          };

          That gives an error because a type alias cannot be redeclared.

        This makes interface useful when
          you intentionally want extension or declaration merging, especially in libraries

  Common rule of thumb

    Use 'interface' for object shapes that may be extended:

      interface User {
        id: number;
        name: string;
      }

      Think:
        This is a contract something must satisfy

    Use 'type' for unions, primitives, tuples, function aliases, and complex combinations:

      type ID = string | number;

      type Status = "idle" | "loading" | "success" | "error";

      type Point = [number, number];

      type ApiResponse<T> =
        | { ok: true; data: T }
        | { ok: false; error: string };

      Think:
        This is a computed, combined, or alternative type

    For normal object shapes, either is fine
      but many teams pick one style for consistency

  Where devs use them “wrong” or awkwardly

    1. Using interface for something that should be a union
        example

          interface Status {
            value: "loading" | "success" | "error";
          }

        Sometimes this is unnecessary. A simpler type is better:

          type Status = "loading" | "success" | "error";

        Use the object version only if you genuinely need an object:

          const status: Status = "loading";

        Instead of:

          const status = { value: "loading" };

    2. Using giant interfaces for everything

        Some devs make huge interfaces:

          interface User {
            id: number;
            name: string;
            email: string;
            password: string;
            createdAt: string;
            updatedAt: string;
            permissions: string[];
            profile: object;
            settings: object;
          }

        Then they use User everywhere: forms, API responses, database records, UI props, etc.

        That often becomes messy.

        A better approach would be:
          type UserPreview = {
            id: number;
            name: string;
          };

          type UserFormInput = {
            name: string;
            email: string;
          };

          type UserFromApi = {
            id: number;
            name: string;
            email: string;
            createdAt: string;
          };

          Different situations often need different types.

    3. Using type intersections where interface extends would be clearer

        This is valid:
          type Admin = User & {
            role: "admin";
          };

        But for simple object extension, this can be clearer:
          interface Admin extends User {
            role: "admin";
          }

        Especially when modeling object-oriented or library-style APIs.

    4. Using interface when they need mapped types

        This kind of thing needs type:
          type OptionalUser = {
            [K in keyof User]?: User[K];
          };

        Or simply:
          type OptionalUser = Partial<User>;

        An interface is not the right tool for that.

  Practical recommendation for most teams
    Use interface for plain object shapes
        interface User {
          id: number;
          name: string;
        }

    Use type for unions, aliases, tuples, mapped types, and complex compositions
        type Status = "loading" | "success" | "error";
        type ID = string | number;
          type Coordinates = [number, number];

    But the most important thing is consistency.
      In real projects, both are acceptable for simple object shapes,
        so it is common to follow the existing style of the codebase.
